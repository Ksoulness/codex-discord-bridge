# Discord 频道状态灯与消息去重设计

## 文档关系

本文扩展并修正以下两份既有设计：

- `2026-07-17-discord-channel-naming-design.md`：保留中文自动命名，新增频道标题状态灯。
- `2026-07-17-discord-turn-progress-design.md`：保留中间状态识别，改为完成和错误后仍保留终态，并补充用户消息去重。

如生命周期或展示规则冲突，以本文为准。

## 目标

1. Discord 频道标题持续反映该对话最近一轮任务的状态。
2. 过程消息末尾显示任务状态，完成、错误或等待授权后仍保留结果。
3. 同一条 Codex 用户输入在 Discord 只镜像一次。
4. 状态更新和标题改名失败时，不影响原有消息、审批和最终回答的镜像。

## 当前问题与根因

### 用户消息重复

同一轮 Codex 用户输入会在 session JSONL 中连续出现两种结构：

1. `response_item` / `message(role=user)`
2. `event_msg` / `user_message`

两条记录拥有相同的轮次语义，但当前事件尾随器按 JSONL 行位置生成不同的 `eventKey` 和 `itemId`。因此它们都能通过 `mirrored_items(thread_id, item_id)` 的唯一键，并被 Discord 各发送一次。

这不是 Discord 客户端重复渲染，也不是用户真的发送了两次。

### 状态灯位置看似变化

用户消息刚出现而 Codex 尚未输出过程消息时，Bridge 只能创建临时状态消息。首条 Codex 过程消息出现后，状态会迁移到该消息末尾并删除临时消息。因此截图中状态灯最后出现在下方 Codex 消息末尾是预期的迁移结果；真正异常的是上方用户消息被镜像了两次。

### 完成后状态消失

旧设计在正常最终回答出现时删除状态后缀，导致用户回看频道时无法判断任务是完成、失败还是被中断。新设计把每轮的终态作为历史记录保留。

## 方案比较

### 方案一：按轮次驱动的状态机与去重（采用）

- 状态以 `(threadId, turnId)` 为边界。
- 每轮只允许一个用户消息锚点和一个状态目标。
- 标题根据该频道最新轮次的状态改色。
- 状态变化由结构化事件触发，标题改名做短时间合并。

优点是身份明确、不会误删用户有意重复发送的相同文字，重启后也能从持久化状态恢复。

### 方案二：定时轮询并统一校正

周期扫描频道和 session 文件，再修正标题与状态。实现直观，但状态延迟较高，快速经历“错误、恢复、完成”时容易漏掉中间状态，也会增加 Discord 请求量。

### 方案三：按相同文字和时间窗口去重

把短时间内内容相同的用户消息视为重复。实现最少，但用户在不同轮次重发同一句话时会被错误吞掉，因此不采用。

## 状态模型

每个普通 Codex 对话频道只以最新轮次控制标题；每一轮仍独立保存自己的消息状态。

| 内部状态 | 标题灯 | 消息状态 | 是否终态 |
| --- | --- | --- | --- |
| `running` | 🟡 | `🟡 状态：进行中` | 否 |
| `reconnecting` | 🟡 | `🟡 状态：正在重连` | 否 |
| `waitingApproval` | 🔴 | `🔴 状态：等待授权` | 否 |
| `networkError` | 🔴 | `🔴 状态：网络错误` | 是，可恢复 |
| `rateLimited` | 🔴 | `🔴 状态：额度或限流` | 是，可恢复 |
| `systemError` | 🔴 | `🔴 状态：系统错误` | 是，可恢复 |
| `completed` | 🟢 | `🟢 状态：已完成` | 是 |

错误后同一轮继续执行时允许从红灯恢复为黄灯，随后正常完成变为绿灯。新一轮开始时，频道标题变为黄灯，但上一轮消息中的绿灯或红灯历史状态保持不变。

## 频道标题状态灯

Discord 界面中的 `#` 是频道图标，不属于频道名。实际频道名格式为：

```text
🟡-清理执行
🔴-清理执行
🟢-清理执行
```

规则如下：

1. 保留既有中文自动命名结果作为基础标题，只替换最前面的状态灯前缀。
2. 新任务开始或错误恢复执行时使用黄灯。
3. 等待授权、网络错误、额度或限流、系统错误时使用红灯。
4. 正常完成后使用绿灯，并保持到下一轮任务开始。
5. 每个频道在 1 至 3 秒内合并连续状态变化，只提交最后一次标题改名。
6. 目标颜色与当前颜色相同时不调用 Discord 改名接口。
7. 旧轮次的迟到事件不能覆盖新轮次标题；比较轮次顺序后才允许更新。
8. Discord 限流或改名失败时记录失败并重试最新目标状态，不回放已经过时的中间颜色。
9. Bridge 重启后从持久化记录恢复最新轮次状态和标题灯。

## 消息末尾状态

### 展示位置

1. 优先附加在当前轮次最近一条 Codex 过程消息末尾。
2. Codex 尚未产生过程消息时，创建一条临时状态消息。
3. 首条过程消息出现后，把状态迁移到该消息末尾，再删除临时状态消息。
4. 最终回答照常独立镜像，不把状态正文重复发送到最终回答中。
5. 正常完成或异常终止后，不再删除状态后缀。
6. 当前轮次没有任何可附着过程消息时，临时状态消息转为永久终态记录。

### 展示格式

进行中：

```text
🟡 状态：进行中
更新时间：1 分钟前
```

完成：

```text
🟢 状态：已完成
更新时间：刚刚
```

错误：

```text
🔴 状态：网络错误
原因：TLS handshake EOF
更新时间：刚刚
```

错误原因只显示一行经过归类、脱敏和截断的摘要，不粘贴 JSON、调用栈、令牌、请求头或整段日志。等待授权不需要原因行。

### 状态判定

可靠性顺序保持为：

1. Codex app-server 和 Desktop IPC 的结构化轮次、线程与审批事件。
2. session JSONL 中带明确 `turnId` 的结构化事件。
3. 能同时绑定当前 `threadId` 和活动轮次的 Desktop 错误事件。

网络错误包括：

- `stream disconnected before completion`
- `error sending request` 且目标为 `backend-api/codex/responses`
- `tls handshake eof`
- `task_complete` 的 `last_agent_message` 为空
- 当前线程进入可关联本轮任务的 `systemError`

额度或限流只认明确失败标志，例如 `429`、`rate_limit_reached` 和 `insufficient_quota`。普通 `token_count`、用量百分比、全局 PubSub 重连以及远程控制 WebSocket 的 `409 Remote app server already online` 均不触发红灯。

## 用户消息去重

### 主规则

普通 Codex 对话中，一个 `(threadId, turnId)` 只允许建立一个用户消息锚点：

1. 第一条有效的 `sessionUserMessage` 建立用户锚点并发送到 Discord。
2. 同一轮后续出现的另一种用户事件结构不再追加消息。
3. 被抑制事件仍推进 session 游标和来源前沿，避免下次轮询或重启再次回放。
4. 实时尾随、启动回放和历史补齐使用同一条判定规则。
5. 不按消息文字全局去重；不同 `turnId` 即使内容完全相同也必须分别显示。

现有真实事件顺序中，包含完整图片和轮次元数据的 `response_item/message(role=user)` 先出现，因此“第一条有效事件优先”会保留信息更完整的版本。

### 缺少 turnId 的回退

缺少 `turnId` 时不使用宽泛的相同文字去重。仅当两条事件满足以下全部条件，才视为同一次输入的双重表示：

- 来自同一个 `threadId`
- 来源类型恰好是一条 `response_item/message(role=user)` 和一条 `event_msg/user_message`
- 规范化正文与附件标识一致
- 时间戳处于极短的相邻窗口内
- session 来源顺序相邻且中间没有任务开始事件

无法可靠确认时宁可保留消息，避免吞掉用户真实输入。

## 事件顺序与并发

1. 每个频道串行处理状态变化和消息锚点更新。
2. 状态记录包含 `threadId`、`turnId`、轮次顺序、状态版本和更新时间。
3. 同一状态的重复事件只刷新必要的相对时间，不重复创建 Discord 消息。
4. `completed` 只有在不存在更高优先级的本轮错误时才能落为绿灯。
5. 错误与完成事件接近到达时，使用结构化终态和事件顺序裁决，不能先显示红灯又被无内容的伪完成覆盖。
6. 新轮次建立后，旧轮次只能更新自己的历史消息，不能再修改频道标题。

## 持久化与恢复

每轮至少保存：

- `threadId` 与 `turnId`
- 用户消息锚点 ID
- 状态目标消息 ID 及其类型
- 当前状态、简短原因和状态版本
- 基础频道名、目标标题灯和最近一次改名时间
- 最新 session 游标和来源顺序

Bridge 重启时：

1. 恢复每个频道最新轮次的标题灯。
2. 保留历史轮次已完成或错误的状态后缀。
3. 活动轮次继续编辑原状态目标，不创建第二条状态消息。
4. Discord 目标消息已被删除时，清除失效 ID；仅活动轮次允许重建临时状态消息。

状态记录随现有线程和轮次保留策略一起清理，不无限增长。

## 失败降级

- 标题改名失败不阻断消息镜像。
- 状态附加、迁移或编辑失败不阻断最终回答。
- Discord 限流时只保留最新待执行标题，不排队重放所有颜色变化。
- 错误原因解析失败时只显示分类状态，不发送原始日志。
- 去重判定异常时记录结构化日志，并优先避免吞掉真实用户消息。

## 测试范围

### 用户消息去重

- 相邻的 `response_item/message(role=user)` 与 `event_msg/user_message` 只发送一条 Discord 用户消息。
- 两种事件顺序颠倒时仍只发送一条。
- 实时尾随、启动回放和历史补齐都不会产生第二条。
- 同一用户在两个不同 `turnId` 重复发送相同文字时保留两条。
- 被抑制事件仍推进游标，Bridge 重启后不会再次出现。

### 状态消息

- 没有过程消息时创建一条临时黄灯状态。
- 首条 Codex 过程消息出现后迁移状态并删除临时消息。
- 完成后保留 `🟢 状态：已完成`。
- 网络、额度和系统异常后保留红灯状态与一行简短原因。
- 等待授权显示红灯，授权后恢复黄灯。
- 错误恢复执行后红灯变黄灯，正常完成后变绿灯。
- `last_agent_message` 为空不能被判为正常完成。
- 普通额度统计、正文关键字和全局重连日志不会误触发状态。

### 频道标题

- 新轮次开始后标题变黄，等待授权或错误变红，正常完成变绿。
- 连续状态在合并窗口内只执行最后一次改名。
- 旧轮次迟到事件不能覆盖新轮次颜色。
- 标题改名失败不影响消息和最终回答。
- Bridge 重启后恢复正确颜色。

## 验收标准

1. 截图中的同一条用户输入在 Discord 只出现一次。
2. 状态迁移后只保留一个状态位置，不出现带状态和不带状态的用户消息副本。
3. 任务执行时频道标题为黄灯，等待授权或发生错误时为红灯，正常完成时为绿灯。
4. 完成和错误状态在任务结束后仍可回看。
5. 用户故意在不同轮次发送相同内容时不被去重。
6. 全量测试通过，Bridge 重启后状态、去重和原有镜像功能均正常。
