# Discord 任务中间状态设计

## 目标

Discord 已经会镜像用户消息和 Codex 最终回答，因此不再重复发送“任务开始”和“任务完成”。本功能只补充两条正常对话之间的任务状态，让用户能判断 Codex 正在工作、等待处理，还是已经遇到异常。

## 展示方式

状态优先附加在当前 Codex 过程消息的末尾，状态变化时编辑这条过程消息，不连续新增消息。只有 Codex 尚未产生任何可附着的过程消息时，才临时创建一条独立状态消息；首条过程消息出现后，把状态迁移到过程消息末尾并删除临时消息。

状态消息示例：

- `🟡 状态：进行中`
- `🟠 状态：等待审批`
- `🔵 状态：正在重连`
- `🔴 状态：网络错误`
- `🔴 状态：额度或限流`
- `🔴 状态：系统错误`

状态行只显示简短状态和更新时间，不重复用户问题、项目名、对话名或完整错误日志，因为 Discord 频道本身已经提供这些上下文。

## 生命周期

1. 用户消息成功镜像到 Discord 后，在已有 Codex 过程消息末尾附加 `🟡 状态：进行中`；若过程消息尚不存在，则创建临时状态消息。
2. 首条 Codex 过程消息出现后，把临时状态迁移到该过程消息末尾，并删除临时消息。
3. 同一轮次出现审批请求时，把末尾状态行改为 `🟠 状态：等待审批`。
4. 审批处理完成并继续执行时，把末尾状态行改回 `🟡 状态：进行中`。
5. 收到当前轮次的可靠重连信号时，改为 `🔵 状态：正在重连`。
6. 收到网络、额度、限流或系统异常时，在末尾显示对应错误状态并保留，便于用户介入。
7. 收到正常最终回答时，只移除过程消息末尾的状态行，过程正文和最终回答都保留；若使用临时状态消息，则删除临时消息。
8. 新轮次开始前发现上一个轮次仍残留状态时，先按其真实终态清理或保留异常，再为新轮次设置状态。

## 状态判定

状态来源按可靠性排序：

1. Codex app-server 和 Desktop IPC 的结构化轮次、线程与审批事件。
2. 本地 session JSONL 中带明确 `turn_id` 的结构化事件。
3. Desktop 日志中同时带明确 `conversationId` 或 `turnId` 的任务级错误事件。

网络错误包括：

- `stream disconnected before completion`
- `error sending request` 且目标为 `backend-api/codex/responses`
- `tls handshake eof`
- `task_complete` 的 `last_agent_message` 为空
- 当前线程进入 `systemError`，并且能关联到本轮任务

额度或限流只认结构化错误中的 `429`、`rate_limit_reached`、`insufficient_quota` 等明确失败标志。普通 `token_count`、`rate_limits.used_percent` 和额度统计不算错误。

## 防止误报

以下内容一律不触发任务状态：

- 用户消息、Codex 回复、命令参数或命令输出中出现“网络错误”等文字。
- 普通 `token_count` 或 `rate_limits` 用量统计。
- 与具体对话无关的 ChatGPT PubSub 重连日志。
- app-server 远程控制 WebSocket 的 `409 Remote app server already online`。
- 浏览器、描述生成、历史记录加载等与当前轮次无关的后台错误。

只有能够绑定当前 Discord 频道对应 `threadId`，并能绑定当前 `turnId` 或当前活动轮次的事件，才能更新状态。

## 持久化与去重

- 为每个线程保存目标消息 ID、目标类型（过程消息或临时消息）、轮次 ID、状态类型和更新时间。
- 同一 `(threadId, turnId)` 的重复事件只编辑目标消息末尾的状态行。
- 同一种错误状态设置冷却时间，避免日志重复导致频繁编辑。
- Bridge 重启后读取已保存状态；活动轮次复用原消息，正常结束的轮次清除旧消息，异常状态继续保留。

## 边界

- 仅作用于普通 Codex 对话频道；子代理线程保持现有行为。
- 不新增 Discord 命令和配置开关。
- 不修改现有用户消息、思考摘要、命令摘要、审批卡片和最终回答格式。
- 不改变 Telegram 脚本，也不让 Telegram 和 Discord 相互依赖。

## 错误处理

- 状态附加、编辑、迁移或清理失败只记录日志，不能阻断正常对话镜像。
- Discord 返回消息 ID 后再写入本地状态库。
- 已保存的 Discord 消息不存在时，清除失效 ID；活动轮次允许重新创建状态消息。
- 错误详情只在本地日志保存，Discord 只显示经过归类的简短状态。

## 测试范围

- 有 Codex 过程消息时，“进行中”附加在过程消息末尾，不创建独立消息。
- 没有 Codex 过程消息时只创建一条临时“进行中”，首条过程消息出现后迁移并删除临时消息。
- 重复用户事件、轮询和回放不会重复创建状态消息。
- 审批请求与审批完成会在同一条消息上切换状态。
- 正常最终回答会移除过程消息末尾状态行，并继续保留过程正文和原有最终回答。
- `last_agent_message` 为空会显示“网络错误”，不能误判为正常完成。
- 断流、请求发送失败和 TLS 握手中断会显示“网络错误”。
- 明确的 429 或额度失败会显示“额度或限流”。
- 普通额度统计、聊天正文、命令输出、全局 PubSub 重连和远程控制 409 不会触发状态。
- Bridge 重启后能恢复活动状态，并清理已正常完成的旧状态。
- 子代理和合成用户事件不会创建普通任务状态。
